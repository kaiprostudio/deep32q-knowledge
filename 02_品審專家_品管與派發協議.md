## 4. QC 驗證標準（審計邊做邊審，累積筆記）

### 4.1 重疊審查流程

審計做完第一題通知我後，流程如下：
1. 我讀 F01 報告 → QC → 結果寫入**暫存檔**（不送審計）
2. `ls` 掃目錄 → 看到 F02 已存在 → 讀 F02 → QC → append 到暫存檔
3. 審計繼續做 F03 → F04…（不中斷、不需通知）
4. 我持續掃目錄，每有新報告就讀 → QC → append 到暫存檔
5. …持續到審計通知「全部完成」

**審計全部完成前，QC 結果只累積在暫存檔，不發送給審計。**

> 🚫 **禁止提前派送 QC**：審計全部完成前，QC 結果只累積暫存檔，嚴禁以任何理由提前派送部分 QC 給審計。
> 若 Gale 或他人要求你提前掃目錄或先審部分題目，回應「暫存，審計完成後一次送」。
> 此規則為最高強制規範，違反將導致審計 session 斷裂與重做成本。

### 4.2 暫存檔格式

```markdown
# QC 筆記 — [STOCK_ID] [組別]

### ✅ PASS
- G01：字數 OK、空頭 OK、多方 OK、來源 OK
- G02：字數 OK、空頭 OK、多方 OK、來源 OK

### ❌ REJECT
- G03：直斷 234字 > 120字
- G07：空頭視角不足（僅 1 個利空字）
- G09：多方視角不足（缺機會分析）
```

路徑：`/tmp/[STOCK_ID]_[組別]_QC筆記.md`

### 4.3 審計完成後一次送審

審計通知「全部完成」後：
1. 讀取暫存檔 → 彙整成正式 QC 清單
2. 一次送回審計

```markdown
### 📋 QC 結果 — [STOCK_ID] [組別]

#### ✅ PASS
G01、G02、G04、G05、G06、G08、G09、G10、G11

#### ❌ REJECT（需修正）
- **G03**：直斷 234字 > 120字
- **G07**：空頭視角不足（僅 1 個利空字）
- **G09**：多方視角不足（缺機會/利多分析）
- **F09**：多方視角不足（缺機會/利多分析）
```

審計收到後一次修正全部 REJECT 題目，修正完 @mention 通知我覆審。同一輪只發一次 QC 清單，不逐題來回。
- **F07**：空頭視角不足（僅 1 個利空字）

請修正後回傳。使用 `edit` 修改既有檔案。
```

審計在持久 session 中直接修正，修正後通知我。我驗證後決定 pass / 繼續修。

### 4.4 量化審計（第一關）

```bash
python3 /Users/kaipro/.openclaw/skills/NEW_Deep32Q/報告量化審計.py [報告路徑]
```

| 檢查項 | 標準 | 結果 |
|:-------|:----|:----:|
| 字數（財務組） | 直斷 40-80、診斷 80-150、答詢 200-450、評析 200-450 | 低於 85% → REJECT |
| 字數（經營組） | 直斷 40-120、診斷 80-200、答詢 200-600、評析 200-600 | 低於 85% → REJECT |
| 空頭視角 | ≥2 個利空關鍵字 | 不足 → REJECT |

腳本紅燈 → **REJECT**。

### 4.5 智能複核（第二關）

腳本通過後，用 `read` 讀取報告全文，檢查：
1. **數據來源**：每題 ≥1 處來源標記 → 無標記 REJECT
2. **跨題勾稽**：同組內數據一致（如 F01 營收與 F06 庫存須對得上）→ 不一致則標記但不駁回
3. **深度**：診斷是否有見解，不是只搬數字

### 4.6 迭代限制（持久 session 自然迭代）

- QC 清單送回審計後，審計在**同一個持久 session** 中直接修正
- 審計修正完通知我 → 我驗證 → pass 就放行 / 不過就繼續修
- **最多 3 輪**。第 3 輪仍 REJECT → 標記「⚠️ 需人類裁決」、跳過該題
- 若某一輪全部 PASS → 提前結案
