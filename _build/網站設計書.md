# Deep32Q 知識庫網站 — 設計書

> 建立於 2026-06-19
> 設計依據：與周一討論之使用者習慣與資訊架構需求

---

## 一、核心設計原則

1. **使用者視角優先**：網站為人類閱讀設計，非為檔案系統設計。Dr32 agent 的目錄結構（Deep32Q知識庫/、審計報告庫/）不對外暴露。
2. **點一次就到內容**：不得有中間目錄頁面強制跳轉。所有報告清單 → 點標題 → 直接顯示內容。
3. **首頁即儀表板**：首頁的首要功能是展示「今天最新的每日報告」，以時間軸形式呈現。
4. **全域全文搜尋**：一個搜尋列於頂欄常駐，搜尋範圍涵蓋所有 `.md` 檔案（每日報告、產業洞察、審計報告）。
5. **離線安全**：網頁是純靜態 HTML，無後端、無資料庫、無 runtime 相依。開 10 年不會壞。

## 二、網站資訊架構

```
首頁（每日報告時間軸）
├─ 展開今天（最新一日所有報告）
│   ├─ 市場情緒
│   ├─ 黃金追蹤
│   ├─ 黃豆追蹤
│   ├─ 白銀追蹤
│   └─ …各領域追蹤
├─ 可摺疊的歷史日期
│
├─ 導航至 [產業洞察]
├─ 導航至 [審計報告]
└─ 頂欄全域搜尋
```

### 三個區塊的定位

| 頁面 | 使用者意圖 | 內容來源 |
|:-----|:----------|:--------|
| **每日報告**（首頁） | 每天打開第一個要看的就是今天的市場變化 | `每日報告/YYYY-MM-DD/`（build.py 會從多個位置收集每日報告統一放到此目錄） |
| **產業洞察** | 對某個產業有興趣時深入了解 | `產業洞察/*/`（含供應鏈拆解，已合併） |
| **審計報告** | 對某隻個股有興趣時查看完整分析 | `審計報告庫/*/` |

## 三、每頁 UI 詳細規格

### 3.1 頂欄（全域）

```
[Deep32Q logo/標題]      [🔍 搜尋知識庫]      [每日報告] [產業洞察] [審計報告]
```

- 搜尋列始終可見
- 搜尋結果以浮動面板顯示，按檔案類型分類（📅每日報告、📁產業洞察、📊審計報告）
- 點擊搜尋結果 → 跳轉至該檔案內容頁面

### 3.2 首頁 — 每日報告時間軸

**視覺結構**：

```
📅 2026-06-19  ▼        ← 預設展開今天

  📊 市場情緒
  [摘要 1~2 行]
  [閱讀全文] → 展開顯示完整內容

  🥇 黃金追蹤  $4,171.20 (-1.50%)
  [摘要 1~2 行]
  [閱讀全文] → 展開顯示完整內容

  🌾 黃豆追蹤  1,122.75 (-0.82%)
  ...

📅 2026-06-18  ▶        ← 預設摺疊，可點開
📅 2026-06-17  ▶
```

**行為**：
- 預設只展開最新的一天（今天或最新有資料的日期）
- 歷史日期均摺疊
- 每份報告顯示：[emoji 圖示] + [標題] + [摘要第一段]
- 點「閱讀全文」→ 在同一頁面 inline 展開完整內容（非跳轉新頁面）
- 展開的內容為完整 .md 渲染後的 HTML

**關於摺疊/展開**：
- 日期層：點擊日期標題行 → 展開/摺疊該日所有報告
- 報告層：點擊「閱讀全文」或報告標題行 → 展開/摺疊該份報告全文
- 兩層互不干擾

## 四、技術架構

### 4.1 建置流程

```
.git/ (master branch)
│
├── Deep32Q知識庫/    ← Dr32 每日寫入
├── 審計報告庫/      ← 審計報告
├── _build/          ← 建置腳本與設計書
│   ├── build.py     ← 主建置腳本
│   ├── template.html← HTML 頁面模板
│   └── style.css    ← 網站樣式
│
├── .github/workflows/deploy.yml  ← CI 觸發
│
build.py 輸出 → _site/ (純靜態 HTML) → gh-pages branch
```

### 4.2 build.py 功能

| 步驟 | 說明 |
|:-----|:-----|
| 1. 掃描 | 掃描 Deep32Q知識庫/ 和 審計報告庫/ 所有 .md 檔案 |
| 2. 分類 | 依目錄路徑分類：每日報告、產業洞察、審計報告 |
| 3. 轉換 | 每份 .md → .html（使用 Python markdown 標準庫） |
| 4. 索引 | 建立全文搜尋索引（search_index.json） |
| 5. 頁面 | 生成首頁、產業洞察列表、審計報告列表、個股頁、報告內容頁 |
| 6. 輸出 | 寫入 _site/ 目錄，含 CNAME、CSS、JS、nav.json、route_map.json |

#### 4.2.1 每日報告來源
build.py 掃描以下位置的 .md 檔案作為每日報告：
- `Deep32Q知識庫/每日報告/YYYY-MM-DD/`（主要路徑）
- `Deep32Q知識庫/市場情緒/`（獨立市場情緒檔案）
- `Deep32Q知識庫/產業洞察/XXX研究/追蹤日誌/`（各產業領域的日誌）

Dr.32 每日寫入後，build.py 會將這些檔案統一歸類到每日報告時間軸。

### 4.3 前端檔案

| 檔案 | 內容 |
|:-----|:-----|
| `_build/template.html` | HTML 模板，所有頁面共用。含 navbar、搜尋、內容區 DOM |
| `_build/style.css` | 完整網站樣式（深色/亮色模式、時間軸、表格、響應式） |
| `_build/search.js` | 前端搜尋引擎，載入 search_index.json 後提供即時全文搜尋 |
| `_build/content.js` | 內容路由器。支援四種路由：home、industries、audit、report。讀取 route_map.json 載入報告 HTML 內容，並從 nav.json 渲染列表頁面 |

### 4.4 部署方式

使用 GitHub Actions，觸發條件：
- `master` branch push（含 Dr32 自動 commit 或手動 commit）
- `workflow_dispatch`（手動觸發）

Actions 流程：
1. Checkout repo
2. 安裝 Python（3.x，預設就有）
3. 執行 `python3 _build/build.py`
4. 上傳 `_site/` 為 Pages artifact
5. 部署至 GitHub Pages

**不需要 npm/node**，純 Python。

## 五、現存問題與解決方案

### 5.1 已移位的檔案（INDEX.md 路徑不一致）

目前 INDEX.md 檔案（位於 Deep32Q知識庫/INDEX.md）是給 Dr32 agent 查閱的，網站不再使用該檔案作為目錄。

**解決方案**：保留 INDEX.md 不刪（Dr32 agent 仍需要），網站完全無視它，由 build.py 掃描實際檔案系統來建立列表。

### 5.2 頁面路由設計

由於是純靜態站（無後端路由），使用 URL hash 或 query parameter 做頁面切換：
- `https://deep32q.kaiproapp.com/` → 首頁（每日報告時間軸）
- `https://deep32q.kaiproapp.com/?p=industries` → 產業洞察
- `https://deep32q.kaiproapp.com/?p=audit` → 審計報告列表
- `https://deep32q.kaiproapp.com/?p=report&file=黃金研究/黃金供需分析` → 某份報告內容

或使用 `#` hash 路由（不發送 HTTP 請求）達到相同效果。

## 六、維護指引

- 修改網站樣式 → 編輯 `_build/style.css`
- 修改頁面佈局 → 編輯 `_build/template.html`
- 修改建置邏輯 → 編輯 `_build/build.py`
- 新增功能 → 更新本設計書後再改程式

所有網站相關的碼和文件全在 `_build/` 資料夾，一次性找到。
