> 建立於 2026-06-19 · 最後更新 2026-06-22
> 設計依據：與周一討論之使用者習慣與資訊架構需求

---

## 一、核心設計原則

1. **使用者視角優先**：網站為人類閱讀設計，非為檔案系統設計。Dr.32 agent 的目錄結構（`Deep32Q知識庫/`、`審計報告庫/`）不對外暴露。
2. **點一次就到內容**：不得有中間目錄頁面強制跳轉。所有報告清單 → 點標題 → 直接顯示內容。
3. **首頁即儀表板**：首頁的首要功能是展示每日報告，以時間軸形式呈現。
4. **全域全文搜尋**：一個搜尋列於頂欄常駐，搜尋範圍涵蓋所有 `.md` 檔案（每日報告、產業洞察、審計報告）。
5. **離線安全**：網頁是純靜態 HTML，無後端、無資料庫、無 runtime 相依。

## 二、網站資訊架構

```
首頁（每日報告時間軸）
├─ 展開今天（最新一日所有報告）
│   ├─ 市場情緒
│   ├─ 黃金追蹤
│   ├─ 黃豆追蹤
│   ├─ 白銀追蹤
│   └─ 各領域追蹤
├─ 可摺疊的歷史日期
│
├─ 導航至 [產業洞察]
├─ 導航至 [審計報告]
└─ 頂欄全域搜尋
```

### 三個區塊的定位

| 頁面 | 使用者意圖 | 內容來源 |
|:-----|:----------|:--------|
| **每日報告**（首頁） | 每天打開第一個要看的就是今天的市場變化 | `Deep32Q知識庫/每日報告/YYYY-MM-DD/`（Dr.32 寫入後，同步腳本複製至此） |
| **產業洞察** | 對某個產業有興趣時深入了解 | `Deep32Q知識庫/產業洞察/*/` |
| **審計報告** | 對某隻個股有興趣時查看完整分析 | `審計報告庫/*/` |

## 三、每頁 UI 詳細規格

### 3.1 頂欄（全域）

```
[Deep32Q logo/標題]      [🔍 搜尋知識庫]      [每日報告] [產業洞察] [審計報告] [管理] [🌙/☀️]
```

- 搜尋列始終可見
- 搜尋結果以浮動面板顯示，按檔案類型分類（📅每日報告、📁產業洞察、📊審計報告）
- 點擊搜尋結果 → 跳轉至該檔案內容頁面
- 右側有深色/亮色模式切換按鈕

### 3.2 首頁 — 每日報告時間軸

**視覺結構**：

```
📅 2026-06-19  ▼        ← 預設展開今天

  📊 市場情緒
  [摘要 1~2 行]
  ▼ [展開顯示完整內容]

  🥇 黃金追蹤  $4,171.20 (-1.50%)
  [摘要 1~2 行]
  ▼ [展開顯示完整內容]

  🌾 黃豆追蹤  1,122.75 (-0.82%)
  ...

📅 2026-06-18  ▶        ← 預設摺疊，可點開
📅 2026-06-17  ▶
```

**行為**：
- 預設只展開最新的一天
- 歷史日期均摺疊
- 每份報告顯示：標題 + 首段摘要
- 點報告標題行 → 在同一頁面 inline 展開完整內容
- 展開的內容為完整 .md 渲染後的 HTML

### 3.3 產業洞察頁

**視覺結構**：

```
產業洞察

📋 PCB研究 (6 份報告)        ▶  ← 預設全部摺疊
💰 黃金研究 (4 份報告)        ▶
🌾 黃豆研究 (25 份報告)       ▶
...
```

**行為**：
- 以產業資料夾為單位，按最新報告日期排序（最新優先）
- 點產業標題 → 展開顯示該產業底下所有文件清單
- 預設全部摺疊
- 點報告標題 → 在當前頁面 inline 展開完整內容（無跳轉）

### 3.4 審計報告頁

**視覺結構**：

```
審計報告

搜尋個股： [_______________]

6239 力成 (29 題)        ▶    ← 點開後顯示經營組/財務組
2313 華通 (29 題)        ▶
2492 華新科 (...)        ▶
...
```

**行為**：
- 預設顯示所有審計過的個股列表（依最新報告日期排序）
- 搜尋框可依股票代碼或公司名稱過濾
- **三層 accordion**：股票 → 經營組/財務組 → 報告標題
- 點報告標題 → 在當前頁面 inline 展開完整內容（無跳轉）

### 3.5 內容閱讀頁

- 用於透過搜尋結果進入的報告（非 inline 展開）
- 左上角有「← 回上頁」按鈕（依來源決定回退目標）
- 中央為完整 Markdown 渲染內容

## 四、技術架構

### 4.1 建置流程

```
.git/ (master branch)
│
├── Deep32Q知識庫/    ← Dr.32 每日寫入
├── 審計報告庫/      ← 審計報告
├── _build/          ← 建置腳本、前端檔案、設計書
│   ├── build.py     ← 主建置腳本（Python，零 npm 依賴）
│   ├── style.css    ← 網站樣式
│   ├── content.js   ← SPA 內容路由器
│   ├── search.js    ← 前端全文搜尋引擎
│   ├── admin/       ← 管理面板
│   └── 網站設計書.md
│
├── CNAME            ← deep32q.kaiproapp.com
├── .github/workflows/deploy.yml  ← CI 觸發
│
build.py 輸出 → _site/ (純靜態 HTML) → GitHub Pages 部署
```

**重要**：本網站由 `_build/build.py`（純 Python）建置。`.vitepress/` 和根目錄的 `index.md`、`package.json`、`build_static.py`、`build_static_v2.py` 均為**未清理的歷史遺留**，不再使用。

### 4.2 build.py 功能

| 步驟 | 說明 |
|:-----|:-----|
| 1. 掃描 | 掃描 `Deep32Q知識庫/` 和 `審計報告庫/` 所有 `.md` 檔案 |
| 2. 分類 | 依目錄路徑與檔名分類：每日報告、產業洞察、審計報告 |
| 3. 轉換 | 每份 `.md` → `.html`（使用 `markdown` Python 套件） |
| 4. 索引 | 建立全文搜尋索引 `search_index.json` |
| 5. 頁面 | 生成首頁（每日報告時間軸）、產業洞察列表、審計報告列表 |
| 6. 輸出 | 寫入 `_site/` 目錄 |

#### 4.2.1 每日報告來源
build.py 從以下位置掃描每日報告：
- `Deep32Q知識庫/每日報告/YYYY-MM-DD/`（主要路徑，Dr.32 同步腳本複製至此）
- `Deep32Q知識庫/市場情緒/YYYY-MM-DD.md`（獨立市場情緒檔案）
- `Deep32Q知識庫/產業洞察/*/追蹤日誌/YYYY-MM-DD.md`（各產業領域的日誌）

### 4.3 前端檔案

| 檔案 | 內容 |
|:-----|:-----|
| `_build/style.css` | 完整網站樣式（深色/亮色模式、時間軸、表格、響應式） |
| `_build/content.js` | SPA 內容路由器，支援四種路由：home、industries、audit、report |
| `_build/search.js` | 前端搜尋引擎，載入 search_index.json 後提供即時全文搜尋 |

### 4.4 部署方式

使用 GitHub Actions，觸發條件：
- `master` branch push（含 Dr.32 自動 commit 或手動 commit）
- `workflow_dispatch`（手動觸發）

Actions 流程：
1. Checkout repo
2. 安裝 Python（`pip install markdown`）
3. 執行 `python3 _build/build.py`
4. 上傳 `_site/` 為 Pages artifact
5. 部署至 GitHub Pages → `deep32q.kaiproapp.com`

**不需要 npm/node**，純 Python。

## 五、資料流與數據來源

### 5.1 資料流關係圖（當前版本）

```
Dr.32 夜間工作（每日 05:00 cron）
  │
  ├─ Step 1~6：搜尋資料 → 寫入各自目錄（.md）
  │   （市場情緒、黃豆、黃金、白銀、光通訊）
  │
  ├─ BOM 轉換（UTF-8 BOM 確保中文不亂碼）
  │
  ├─ 每日報告同步腳本
  │   （複製各領域報告到 每日報告/YYYY-MM-DD/）
  │
  ├─ 維護
  │   ├─ 更新 INDEX.md
  │   ├─ 更新 LOG.md
  │   └─ git add + commit + push
  │
  └─ ⚠️ Telegram 文字摘要：由 OpenClaw framework 自動 deliver
     （不再上傳 .md 實體檔案附件，網站上已有）
          │
          ▼
     GitHub Actions (push 觸發)
       ├─ python3 _build/build.py
       │   （掃描 .md → 分類 → MD→HTML → 索引 → 頁面生成）
       └─ deploy-pages → deep32q.kaiproapp.com
```

### 5.2 頁面路由設計

純靜態站使用 URL query parameter 做頁面切換（SPA）：

| URL | 頁面 |
|:----|:----|
| `https://deep32q.kaiproapp.com/` | 首頁（每日報告時間軸） |
| `https://deep32q.kaiproapp.com/?p=industries` | 產業洞察 |
| `https://deep32q.kaiproapp.com/?p=audit` | 審計報告列表 |
| `https://deep32q.kaiproapp.com/?p=report&f=路徑` | 某份報告內容 |

路由由 `_build/content.js` 處理，無後端參與。

### 5.3 檔案名相容性

GitHub Pages 對長中文 URL 有 Unicode normalization 問題（NFD vs NFC）導致 404。解決方案：
- `reports/` 下的報告檔案使用 MD5 hash[:12] 純 ASCII 檔名
- `route_map.json` 記錄 route → hash 路徑的映射
- 前端 content.js 透過 `routeMap[route]` 取得正確路徑

## 六、維護指引

- 修改網站樣式 → 編輯 `_build/style.css`
- 修改 SPA 路由 → 編輯 `_build/content.js`
- 修改搜尋邏輯 → 編輯 `_build/search.js`
- 修改建置邏輯 → 編輯 `_build/build.py`
- 新增功能 → 先更新本設計書後再改程式

所有網站相關的碼和文件全在 `_build/` 資料夾，一次性找到。
